## 简介

中文描述生成 Git 命令，配合本地规则引擎做风险检测，延续⑫⑬⑭的"本地规则+AI辅助"路线。Git 的破坏性命令（`push --force`、`reset --hard` 等）比 SQL 的 DELETE 更直观、更容易讲清楚危险性，核心定位是帮开发者在执行高危命令前看清后果。

## 功能

### Tab1：描述生成

- 中文描述需求 → AI 生成 1-3 个候选命令方案
- **意图分流卡片**：当描述存在多种合理实现方式时（如"撤销commit"可能对应 `reset --soft`/`reset --hard`/`revert` 三种完全不同的语义），AI 会把每种方式作为独立卡片列出，而非强行给单一答案，避免选择恐惧和误操作
- 风险等级和高危参数高亮**始终由本地规则引擎重新计算**，不采用 AI 自报的风险判断——AI 只负责生成候选命令和场景说明，安全性判断完全交给确定性的本地规则
- 顶部"状态假设开关"联动调整候选卡片的风险等级

### Tab2：场景速查（零AI，本地静态数据）

- 覆盖五大高频分类：撤销/回退、分支管理、远程同步、历史修改、暂存区操作
- 每条给出：命令模板 + 一句话说明 + 风险等级标注 + 一键复制
- **实时模糊搜索**：支持中文关键词（如"回退"）和英文关键词（如"amend"）搜索，不引入完整拼音引擎，而是为每条预设场景手工标注高频搜索词，命中即显示，故障排查场景下不用翻目录
- 现代命令（`git switch`/`git restore`）标注版本兼容性徽章，避免在老旧 Git 版本的服务器上复制后报错

### Tab3：风险检测器（零AI，本地规则匹配）

- 粘贴/输入任意 git 命令，本地规则库实时检测高危模式
- **Token 级高危参数高亮**：只对命令中的危险参数本身（如 `--force`、`--hard`）应用醒目样式，命令主干保持常规颜色，避免全行标红的视觉疲劳
- 检测到高危模式时给出"为什么危险"+"更安全的替代方案"，并提供**一键替换**按钮直接用安全命令覆盖输入框
- 兼容性徽章同样适用

### 状态假设开关（Tab1 和 Tab3 共用）

- `[ ] 假设该分支已经推送到远程共享仓库`
- `[ ] 假设当前本地工作区有未提交的临时代码`
- 两个开关的勾选状态在 Tab1 和 Tab3 之间保持同步
- 勾选后，本地规则引擎会动态将相关命令的风险等级升级（如 `push --force` 在"已推送远程"勾选后从黄色警告升级为黑红色"极危"），并注入具体的升级提示文案

## 技术要点

- **AI 输出不被信任为安全判断依据**：这是核心防御设计。AI 生成候选命令后，风险等级、高亮位置、是否触发升级，全部交给本地规则引擎重新计算，即使 AI 对某条命令的风险判断有误，最终展示给用户的安全等级依然是确定性的、已经过测试验证的本地规则结果
- **Token 位置定位**：规则匹配函数返回危险参数在原始命令字符串中的精确字符位置（start/end），用于渲染层做字符级高亮，复用了⑫正则生成器"高亮联动定位"的技术思路
- **负向先行断言的精确性**：`push --force` 和 `push --force-with-lease` 必须被严格区分，正则中的负向断言位置经过实测验证（`\b` 词边界在判断"force"后是否紧跟"-with-lease"时容易误判，需要把否定前瞻直接挂在目标词后面而不是放在 `\b` 之后）
- **本地搜索关键词标注**：不引入拼音转换库，用人工标注的中英文关键词数组覆盖高频搜索意图，在场景数量可控的范围内（25条左右）是更轻量且足够准确的方案

## 不做的（v1 明确排除）

- 不做真实 git 操作执行（纯命令生成/解释/检测，不连接用户本地仓库，也无法连接——浏览器环境本身就做不到）
- 不做 commit message 规范生成器（可作为独立小功能，不并入 v1）
- 不做图形化分支可视化（工作量大，跟"记不住命令"的核心痛点关系不大）

## 已知限制

- 风险检测基于命令文本模式匹配，无法感知用户的真实仓库状态——这正是"状态假设开关"存在的原因：把无法自动感知的局限转化为用户主动声明的交互
- 规则库覆盲范围有限，未收录的命令模式不会触发任何风险提示，工具在无匹配时会明确说明"基于本地规则库，不能替代对命令本身含义的理解"
- AI 生成功能依赖 jotarou.com API Key，场景速查和风险检测两个核心功能完全不需要 Key
